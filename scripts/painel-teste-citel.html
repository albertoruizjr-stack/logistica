Análise · contrato de dados
Mapear quais campos do JSON viram saída do "Buscar pedido"
Tabela de origem definida: cada coluna da tela aponta para um caminho exato no JSON da Citel. Decisão importante: o endereço de entrega da Citel já vem com o número embutido no campo endereco (ex: "RUA X, 1200"), então não preciso reconstruir nada.
ver tabela de mapeamento 
// Campo na tela → caminho no JSON da Citel

nome cliente       →  pedido.cliente.nome
telefone           →  pedido.cliente.telefone1
celular            →  pedido.cliente.telefoneCelular
CPF/CNPJ           →  pedido.cliente.numeroDocumento
email              →  pedido.cliente.email

endereço cliente   →  pedido.cliente.{endereco, numero, bairro, cep, cidade}
endereço entrega   →  pedido.enderecoEntrega.{endereco, bairro, cep, cidade}

qtd. de itens      →  pedido.itens.length           // = 2 no caso do PD 717081
peso bruto total   →  pedido.pesoBruto              // = 3.48 kg
valor total        →  pedido.valorContabil          // = R$ 114,00

já faturado        →  pedido.jaFaturado             // boolean
cancelado          →  cancelado                     // boolean na raiz

Implementação · serviço
Reescrever services/citel.service.ts com a API correta
Serviço reescrito mantendo as funções antigas de estoque e produto intactas. Três decisões importantes: (1) duas funções de autenticação coexistem — uma por token para estoque, outra Basic Auth para pedido; (2) o response bruto entra num cache de 8 segundos indexado por pedido+loja, então fetchPedidoCabecalho e fetchPedidoItens compartilham a mesma chamada HTTP; (3) o parser separa o campo endereco "RUA X, 1200" em logradouro e número, e trata o bug da API que retorna {cancelado:true, pedido:null} como "não achei" em vez de cancelamento real.
